# CloudZero Agent Monitoring Infrastructure

This document describes the monitoring resources shipped with the CloudZero
Agent Helm chart: what they are, what they cover, what they don't cover, and
how they were validated.

## Overview

The chart provides two categories of monitoring integration:

1. **Prometheus `prometheus.io/*` annotations** on all Services (always enabled).
   These allow standard Prometheus installations using `kubernetes_sd_configs`
   to auto-discover and scrape CloudZero Agent metrics without any CRDs.

2. **Prometheus Operator CRDs** (opt-in via `components.monitoring.enabled`).
   When enabled, the chart creates `ServiceMonitor` and `PrometheusRule`
   resources that the Prometheus Operator automatically picks up.

These resources are designed to be useful regardless of the customer's
monitoring stack. The `ServiceMonitor` and `PrometheusRule` CRDs are the
standard interoperability format understood by the Prometheus Operator, but
also by compatible tools like Victoria Metrics Operator, Datadog (via its
Prometheus integration), Grafana Agent, and others.

## Configuration

```yaml
components:
  monitoring:
    # null = auto-detect (install CRDs if Prometheus Operator is present)
    # true = always install CRDs (fails if Prometheus Operator absent)
    # false = never install CRDs (default while feature is being validated)
    enabled: false

    # Override namespace for CRDs (default: same as agent namespace)
    namespace: ""

    # Additional labels for CRD discovery (e.g., release: kube-prometheus-stack)
    labels: {}

    # Use CA secret for webhook TLS verification instead of insecureSkipVerify
    sharedSecret: false
```

## ServiceMonitors

Four `ServiceMonitor` resources are created, one per scrape target:

| ServiceMonitor Name                 | Target           | Port                     | Protocol | Metrics                                                                                                               |
| ----------------------------------- | ---------------- | ------------------------ | -------- | --------------------------------------------------------------------------------------------------------------------- |
| `{release}-cz-server`               | Prometheus agent | 9090 (`http-metrics`)    | HTTP     | `prometheus_remote_storage_*`, `prometheus_sd_*`, process metrics                                                     |
| `{release}-cz-aggregator-collector` | Collector        | 8080 (`metrics`)         | HTTP     | `metrics_received_total`, `metrics_received_cost_total`, `http_request_duration_seconds`                              |
| `{release}-cz-aggregator-shipper`   | Shipper sidecar  | 8081 (`shipper-metrics`) | HTTP     | `shipper_run_fail_total`, `shipper_handle_request_success_total`, `shipper_file_upload_error_total`, `shipper_disk_*` |
| `{release}-cz-webhook`              | Webhook server   | 8443 (`http`)            | HTTPS    | `czo_webhook_types_total`, `remote_write_*`, `function_execution_seconds`                                             |

### Job and Endpoint Labels

The Prometheus Operator assigns the `job` label from the **Service name**, not
the ServiceMonitor name. Because the collector and shipper are both behind the
same `{release}-cz-aggregator` Service, they share the same `job` label. The
`endpoint` label distinguishes them:

- Collector: `job="{release}-cz-aggregator", endpoint="metrics"`
- Shipper: `job="{release}-cz-aggregator", endpoint="shipper-metrics"`

The alert expressions use both `job` and `endpoint` labels to select the
correct target. The `job` values are generated by the same Helm template
helpers that generate the Service names, so they match regardless of the
Helm release name.

### Webhook TLS

The webhook serves all endpoints (admission webhook + metrics) over HTTPS on
port 8443 using a self-signed certificate (or cert-manager-issued cert). By
default, the ServiceMonitor uses `insecureSkipVerify: true` to scrape it.

Setting `components.monitoring.sharedSecret: true` switches to proper CA
verification by referencing the webhook's TLS Secret. This requires the
Prometheus Operator to have RBAC to read Secrets in the agent's namespace.
See the `values.yaml` comments on `sharedSecret` for full details.

## PrometheusRule Alerts

Fourteen alert rules are defined in a single `PrometheusRule` resource,
organized into three categories.

### Critical Alerts

These indicate the agent is broken and not collecting or delivering data.

| Alert                            | Metric                                             | For                    | Condition             | What It Catches                                                                                                                                                                                                                                  |
| -------------------------------- | -------------------------------------------------- | ---------------------- | --------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `CloudZeroAgentDown`             | `up` / `absent(up)`                                | 5m                     | target down or absent | Prometheus agent pod unreachable or not running. No cluster metrics are being collected. Uses `absent()` to detect pods that are unschedulable or completely missing, not just unhealthy.                                                        |
| `CloudZeroWebhookDown`           | `up` / `absent(up)`                                | 5m                     | target down or absent | Webhook pod unreachable or not running. Kubernetes resource metadata not being captured.                                                                                                                                                         |
| `CloudZeroCollectorNoMetrics`    | `metrics_received_total`                           | 15m                    | `rate == 0`           | Collector is running but not receiving remote-write data from the Prometheus agent.                                                                                                                                                              |
| `CloudZeroAggregatorDown`        | `up` / `absent(up)`                                | 5m                     | target down or absent | Aggregator pod (collector + shipper) unreachable or not running. No metrics are being collected or uploaded.                                                                                                                                     |
| `CloudZeroShipperUploadFailures` | `shipper_run_fail_total`                           | 3.1x `costMaxInterval` | `rate > 0`            | Shipper is failing to upload files. Threshold is dynamic: 3.1x the configured `aggregator.database.costMaxInterval` (default 30m = 93m). The `error_status_code` label indicates the failure category (`err-unauthorized`, `err-network`, etc.). |
| `CloudZeroWebhookNoEvents`       | `czo_webhook_types_total`                          | 30m                    | `rate == 0`           | Webhook is running but receiving zero admission requests. Most dangerous silent failure -- typically indicates a TLS certificate mismatch after cert rotation.                                                                                   |
| `CloudZeroAgentOOMKilled`        | `kube_pod_container_status_last_terminated_reason` | 1m                     | `reason="OOMKilled"`  | A container was killed by the kernel OOM killer. The memory limit is too low for the workload.                                                                                                                                                   |

### Warning Alerts

These indicate degraded operation that may escalate.

| Alert                                 | Metric                                                                      | For                    | Condition             | What It Catches                                                                                                                                                                                                 |
| ------------------------------------- | --------------------------------------------------------------------------- | ---------------------- | --------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CloudZeroShipperNoSuccessfulUploads` | `shipper_handle_request_success_total`                                      | 1.5x `costMaxInterval` | `rate == 0`           | No files successfully uploaded. Threshold is dynamic: 1.5x the configured `aggregator.database.costMaxInterval` (default 30m = 45m). Broader than the upload failures alert -- catches any delivery problem.    |
| `CloudZeroWebhookServerHighLatency`   | `function_execution_seconds_bucket`                                         | 10m                    | `p99 > 500ms`         | Server-side webhook processing latency is high. Does NOT include network/TLS overhead. For end-to-end latency, monitor `apiserver_admission_webhook_admission_duration_seconds` from the Kubernetes API server. |
| `CloudZeroWebhookRemoteWriteFailures` | `remote_write_failures_total`                                               | 15m                    | `rate > 0`            | Webhook cannot push resource metadata to the CloudZero API.                                                                                                                                                     |
| `CloudZeroRemoteWriteLag`             | `prometheus_remote_storage_queue_*`                                         | 10m                    | `lag > 300s`          | Remote-write queue is more than 5 minutes behind. Data is being scraped but not delivered to the collector.                                                                                                     |
| `CloudZeroAgentHighMemoryUsage`       | `container_memory_working_set_bytes` / `kube_pod_container_resource_limits` | 10m                    | `> 85%`               | A container is near its memory limit. Uses `on(namespace, pod, container)` join to correctly match metrics from different sources (cAdvisor and kube-state-metrics).                                            |
| `CloudZeroAgentContainerRestarting`   | `kube_pod_container_status_restarts_total`                                  | 15m                    | `> 2 restarts in 15m` | A container is in a restart loop (CrashLoopBackOff, OOM kills, config errors, missing secrets).                                                                                                                 |

### Data Quality Alerts

These detect scenarios where the agent is running but the data it collects
is insufficient for accurate cost attribution.

| Alert                              | Metric                              | For | Condition                     | What It Catches                                                                                                                                                                                                                                                                             |
| ---------------------------------- | ----------------------------------- | --- | ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CloudZeroMissingContainerMetrics` | `container_cpu_usage_seconds_total` | 30m | container-level labels absent | cAdvisor is emitting metrics but without `container` and `image` labels. Only node/pod-level aggregates are available. This happens when a Docker-based CRI doesn't emit container-level rollups, causing CPU cost attribution to produce zeros. See `docs/metric-level-classification.md`. |

### Dynamic Shipper Alert Thresholds

The shipper alert thresholds are not hardcoded. They are dynamically calculated
from the `aggregator.database.costMaxInterval` Helm value (default `30m`),
which controls how often the collector flushes metric files for the shipper to
upload.

The Helm chart parses the Go duration string (which may be compound, e.g.,
`1h30m`), converts it to total seconds, multiplies by the alert's scaling
factor, and outputs the result as an integer seconds value (e.g., `2700s`).
This format is valid for both Prometheus PromQL range selectors and
PrometheusRule `for` durations.

| Alert                                           | Multiplier | Default (30m) | If 5m | If 1h  | If 1h30m |
| ----------------------------------------------- | ---------- | ------------- | ----- | ------ | -------- |
| `CloudZeroShipperNoSuccessfulUploads` (Warning) | 1.5x       | 2700s (45m)   | 450s  | 5400s  | 8100s    |
| `CloudZeroShipperUploadFailures` (Critical)     | 3.1x       | 5580s (93m)   | 930s  | 11160s | 16740s   |

This ensures that customers who configure faster flush intervals (e.g., `5m`
for dev clusters) get proportionally faster alerts, while customers with slower
intervals don't get false positives from normal jitter.

### What the Alerts Do NOT Cover

- **End-to-end webhook latency** as seen by the Kubernetes API server. The
  `CloudZeroWebhookServerHighLatency` alert only measures internal processing
  time. For full latency including network and TLS overhead, customers should
  monitor `apiserver_admission_webhook_admission_duration_seconds_bucket`
  from the API server's own metrics.

- **Disk pressure on the aggregator volume.** The shipper emits
  `shipper_disk_current_unsent_file` and `shipper_disk_pressure_level` gauges,
  but there is no alert for disk space exhaustion. Customers with constrained
  PVCs may want to add one.

- **Individual scrape target failures.** If the Prometheus agent can't reach a
  specific kubelet or cAdvisor endpoint, no alert fires. Monitor
  `prometheus_sd_discovered_targets` and `up` for specific scrape jobs.

- **CloudZero platform-side issues.** Upload failures are detected
  (`CloudZeroShipperUploadFailures`), but issues occurring after successful
  upload are outside the agent's visibility.

### Counter Metrics That Appear Only After First Event

Several alert metrics are Prometheus counters that only appear in `/metrics`
output after their first event. This is normal Prometheus behavior and is
documented in the "Monitoring the CloudZero Agent" wiki. On a fresh
installation, the following may be absent until their trigger condition occurs:

- `shipper_run_fail_total` -- appears after first upload failure
- `shipper_handle_request_success_total` -- appears after first successful upload
- `remote_write_failures_total` -- appears after first remote-write failure

Alerts that reference absent metrics evaluate to "no data" rather than firing,
which is the correct behavior (absence of the failure counter means no failures
have occurred).

### Alert Firing Behavior

Validated using multiple test scenarios on the `bach` cluster:

**Scenario 1: Invalid API key (natural failure path)**

| Alert                                 | State              | Notes                                                     |
| ------------------------------------- | ------------------ | --------------------------------------------------------- |
| `CloudZeroShipperUploadFailures`      | **Pending/Firing** | Correctly detected `error_status_code="err-unauthorized"` |
| `CloudZeroWebhookNoEvents`            | **Firing**         | Webhook pod on unhealthy node; no admission events        |
| `CloudZeroWebhookRemoteWriteFailures` | **Firing**         | Webhook unable to push metadata                           |
| `CloudZeroRemoteWriteLag`             | **Pending**        | Remote-write queue falling behind                         |

**Scenario 2: Unschedulable pod (memory request set to 64Gi)**

| Alert                | State       | Notes                                                                                                                                                                                                                           |
| -------------------- | ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CloudZeroAgentDown` | **Pending** | `absent(up{...})` correctly detected missing target. Original `up == 0` expression would NOT have fired because the metric becomes absent (not zero) when a pod disappears entirely. Fixed to use `up == 0 or absent(up{...})`. |

**Scenario 3: Memory pressure (memory limit set to 65Mi vs 73.7Mi working set)**

| Alert                               | State       | Notes                                                                                                                                                                                          |
| ----------------------------------- | ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CloudZeroAgentHighMemoryUsage`     | **Pending** | Memory ratio at 93.9%. Required `on(namespace, pod, container)` join -- without it, the division silently produced zero results due to label mismatch between cAdvisor and kube-state-metrics. |
| `CloudZeroAgentContainerRestarting` | **Pending** | Container restarting from memory pressure. Caught restart loop alongside the memory alert.                                                                                                     |

**Scenario 4: OOM kill (memory limit set to 30Mi vs 73.7Mi working set)**

| Alert                               | State       | Notes                                                                                                                                                                       |
| ----------------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CloudZeroAgentOOMKilled`           | **Firing**  | `kube_pod_container_status_last_terminated_reason{reason="OOMKilled"}` correctly detected OOM kill within 1 minute. Standard KSM metric, no configuration changes required. |
| `CloudZeroAgentContainerRestarting` | **Pending** | Caught the CrashLoopBackOff from repeated OOM kills.                                                                                                                        |
| `CloudZeroAgentDown`                | **Pending** | Server target gone due to crash loop.                                                                                                                                       |

**Scenario 5: Collector starved (server + webhook scaled to 0)**

| Alert                         | State       | Notes                                                                                                                                                                                                                                   |
| ----------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `CloudZeroCollectorNoMetrics` | **Pending** | Correctly detected that `rate(metrics_received_total[10m]) == 0`. Note: scaling only the server to 0 is insufficient because the webhook also remote-writes to the collector. Both data sources must be stopped for this alert to fire. |

**Alerts validated as correctly inactive during normal operation:**

| Alert                               | Notes                                                                                                           |
| ----------------------------------- | --------------------------------------------------------------------------------------------------------------- |
| `CloudZeroWebhookServerHighLatency` | p99 latency well below 500ms threshold. Would require sustained load to trigger.                                |
| `CloudZeroMissingContainerMetrics`  | Container-level cAdvisor metrics present (bach uses containerd CRI). Would require Docker-based CRI to trigger. |

**Alerts with intentional design limitations:**

| Alert                                 | Notes                                                                                                                                                                                                                                                                                                                                                                                            |
| ------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `CloudZeroShipperNoSuccessfulUploads` | Detects "was working, then stopped" -- NOT "never worked at all." When the success counter has never been incremented, it is absent in Prometheus and `rate()` returns no data rather than 0. This is intentional: the "never worked" case is covered by `CloudZeroShipperUploadFailures` (which fires on upload errors) and the OOM/restart/down alerts (which catch container-level failures). |

### Bugs Found and Fixed During Testing

1. **`absent()` bug in AgentDown/WebhookDown**: When a pod is completely
   absent (unschedulable, scaled to 0), the `up` metric becomes absent rather
   than zero. The original `up == 0` expression never fires. Fixed to use
   `up == 0 or absent(up{...})`.

2. **Label join bug in HighMemoryUsage**: `container_memory_working_set_bytes`
   (from cAdvisor/kubelet) and `kube_pod_container_resource_limits` (from
   kube-state-metrics) have different label sets. Without an explicit
   `on(namespace, pod, container)` join, Prometheus's implicit matching
   silently produces zero results. Fixed by adding the explicit join.

3. **Job label mismatch in collector/shipper alerts**: The Prometheus Operator
   assigns the `job` label from the Service name, not the ServiceMonitor name.
   Both collector and shipper share the aggregator Service, so they get the
   same `job` label. Fixed by adding `endpoint` label selectors to
   distinguish them.
